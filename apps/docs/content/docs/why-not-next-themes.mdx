---
title: Why not next-themes?
description: A comparison of @wrksz/themes and next-themes.
---

`@wrksz/themes` is a near drop-in replacement for `next-themes`. It fixes every known bug and adds missing features. The API is identical - migrating requires changing one import line.

<Callout type="warn">
  `next-themes` is slowly maintained - 43 open issues and 16 open PRs as of March 2026, with React 19 compatibility bugs still unresolved in the latest release.
</Callout>

## Bug fixes

### React 19 script warning

`next-themes` renders an inline `<script>` inside a Client Component, which triggers a React 19 warning. `@wrksz/themes/next` uses `useServerInsertedHTML` to inject the script outside the React component tree, eliminating the warning entirely.

### `__name` minification bug

Production builds that minify function names break `next-themes` internals. Fixed.

### Stale theme in React 19 Activity and `cacheComponents`

`next-themes` uses `useState` + `useContext`, which causes stale theme values when React's `Activity` or `cacheComponents` suspend and resume a subtree. `@wrksz/themes` uses `useSyncExternalStore` with a per-instance store, which always returns the latest value.

## Missing features

### Multiple classes per theme

Both libraries support space-separated values in the `value` map. The bug is in removal: `next-themes` removes only the top-level theme key as a single class, so switching from a multi-class theme leaves stale classes in the DOM.

`@wrksz/themes` flattens all mapped values before removing them:

```typescript
for (const attr of attrs) {
  if (attr === "class") {
    const toRemove = themes.flatMap((t) =>
      (valueMap?.[t] ?? t).split(" "),
    );
    el.classList.remove(...toRemove);
    el.classList.add(...attrValue.split(" "));
  } else {
    el.setAttribute(attr, attrValue);
  }
}
```

This means switching from `"dark high-contrast"` to `"light"` correctly removes both `dark` and `high-contrast`.

### Nested providers

`next-themes` uses a single global context - nested providers share state. `@wrksz/themes` creates a per-instance store so each provider is fully independent.

### `sessionStorage` and `cookie` storage

```tsx
<ThemeProvider storage="sessionStorage">
<ThemeProvider storage="cookie">
```

`storage="cookie"` is especially useful with `@wrksz/themes/next` - the provider reads the cookie server-side automatically, eliminating SSR theme flash without any manual boilerplate.

### Disable storage

```tsx
<ThemeProvider storage="none">
```

### `meta theme-color` support

Updates `<meta name="theme-color">` automatically when the theme changes. Useful for Safari and PWAs:

```tsx
<ThemeProvider themeColor={{ light: "#ffffff", dark: "#0a0a0a" }}>
```

### Server-provided theme

Initialize the theme from a server-side source (database, session, cookie) on every mount:

```tsx
<ThemeProvider
  initialTheme={userTheme}
  onThemeChange={saveUserTheme}
>
```

### Always follow system preference

`next-themes` has no way to force the theme to always track the OS setting. `@wrksz/themes` adds `followSystem` - it ignores any stored value on mount and always applies `prefers-color-scheme`:

```tsx
<ThemeProvider followSystem>
```

Unlike `defaultTheme="system"` (which stores the resolved value after first visit), `followSystem` never reads from storage.

### `disableTransitionOnChange` per property

`next-themes` only supports `disableTransitionOnChange={true}` which disables all transitions. `@wrksz/themes` accepts a CSS `transition` string to suppress only specific properties:

```tsx
// Disable all transitions (same as next-themes)
<ThemeProvider disableTransitionOnChange>

// Suppress only color properties, keep transform/opacity transitions
<ThemeProvider disableTransitionOnChange="background-color 0s, color 0s, border-color 0s">
```

The style is also injected by the inline script on initial page load - not just when the user switches themes. This means there is no transition flash on first paint either.

### Read theme outside React

`getTheme()` reads the current theme from a cookie without React context - useful in proxy functions or layouts where you need the theme before rendering:

```ts title="proxy.ts"
import { getTheme } from "@wrksz/themes/next";

export function proxy(request: Request) {
  const theme = getTheme(request, { defaultTheme: "dark" });
}
```

```tsx title="layout.tsx"
const theme = await getTheme({ defaultTheme: "dark" });
```

### Generic types

```tsx
const { theme, setTheme } = useTheme<"light" | "dark" | "high-contrast">();
```
